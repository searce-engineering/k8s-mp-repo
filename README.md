# GCP Marketplace GKE Terraform POC

A minimal proof-of-concept for publishing a containerized application to Google Cloud Marketplace using the **Terraform-based GKE integration**. The application image and Helm chart both live in a single Artifact Registry repository.

> **Note:** This POC uses the modern Terraform-based GKE integration, not the legacy Anthos/Kubernetes Deployer model. The Marketplace actuation engine executes standard Terraform directly, eliminating the need for a custom Deployer container.

## Directory Structure

```
tf-mp-poc/
├── README.md
├── py-app/                        # Python application
│   ├── main.py                    # HTTP server with /healthz endpoint
│   └── Dockerfile                 # python:3.12-slim container
├── chart/                         # Helm chart
│   ├── Chart.yaml
│   ├── values.yaml
│   └── templates/
│       ├── deployment.yaml        # Includes readiness and liveness probes
│       ├── service.yaml
│       └── application.yaml       # Required app.k8s.io/v1beta1 CRD
├── scripts/
│   ├── annotate-image.sh          # Applies Marketplace OCI annotation to an image
│   └── package-and-upload.sh      # Zips terraform/ and uploads to GCS
└── terraform/                     # Marketplace Terraform module (what gets zipped)
    ├── main.tf
    ├── variables.tf
    ├── outputs.tf
    ├── versions.tf
    ├── schema.yaml                 # Image-to-variable mapping for Marketplace
    ├── README.md                   # Required by CFT CLI for metadata generation
    ├── metadata.yaml               # Generated by CFT CLI
    ├── metadata.display.yaml       # Generated by CFT CLI
    └── test/
        └── setup/
            ├── iam.tf              # IAM roles read by CFT to populate metadata
            └── main.tf             # GCP services read by CFT to populate metadata
```

## Prerequisites

- Google Cloud SDK (`gcloud`) authenticated with your publisher project
- Docker with `buildx` support
- Helm 3 CLI
- `crane` CLI — [install guide](https://github.com/google/go-containerregistry/tree/main/cmd/crane)
- `cft` CLI — [install guide](https://cloud.google.com/foundation-toolkit/docs/cft-cli)
- Terraform >= 1.4

## Step-by-Step Guide

Set your variables once and reuse them throughout:

```bash
export PROJECT_ID="searce-cloud-products"
export REGION="us"
export REPO_NAME="gcp-mp-poc-tf-k8s"
export REGISTRY="${REGION}-docker.pkg.dev/${PROJECT_ID}/${REPO_NAME}"
export SERVICE_NAME="services/gcp-mp-poc-tf-k8s.endpoints.${PROJECT_ID}.cloud.goog"
export VERSION="1.0.0"
export MINOR_VERSION="1.0"
```

---

### Step 1 — Create Artifact Registry Repository

```bash
gcloud artifacts repositories create $REPO_NAME \
    --repository-format=docker \
    --location=$REGION \
    --description="GKE Marketplace Terraform POC" \
    --project=$PROJECT_ID
```

---

### Step 2 — Build and Push the Application Image

```bash
docker buildx build --platform=linux/amd64 \
    --provenance=false --sbom=false --no-cache \
    -t ${REGISTRY}/py-app:${VERSION} ./py-app

docker tag ${REGISTRY}/py-app:${VERSION} ${REGISTRY}/py-app:${MINOR_VERSION}

docker push ${REGISTRY}/py-app:${VERSION}
docker push ${REGISTRY}/py-app:${MINOR_VERSION}
```

---

### Step 3 — Package and Push the Helm Chart as an OCI Artifact

The Helm chart is pushed as a proper OCI artifact using `helm push`. This allows the Terraform `helm_release` resource to pull it natively without any shell extraction or crane tooling at plan time.

```bash
# Authenticate Helm to Artifact Registry
helm registry login ${REGION}-docker.pkg.dev \
    --username oauth2accesstoken \
    --password $(gcloud auth print-access-token)

# Package the chart
helm package ./chart --version ${VERSION} --app-version ${VERSION}

# Push to Artifact Registry
helm push sample-app-${VERSION}.tgz oci://${REGISTRY}

# Tag the minor version track
crane tag ${REGISTRY}/sample-app:${VERSION} ${MINOR_VERSION}
```

---

### Step 4 — Annotate All Image Tags

Every tag of every image must carry the Marketplace service name annotation. Without it the Marketplace entitlement system will block image pulls.

> The `SERVICE_NAME` is generated by the Producer Portal when you create the product listing. Retrieve it from the portal before running these commands.

```bash
# Annotate application image
crane mutate ${REGISTRY}/py-app:${VERSION} \
    --annotation com.googleapis.cloudmarketplace.product.service.name=${SERVICE_NAME}

crane mutate ${REGISTRY}/py-app:${MINOR_VERSION} \
    --annotation com.googleapis.cloudmarketplace.product.service.name=${SERVICE_NAME}

# Annotate Helm chart OCI artifact
crane mutate ${REGISTRY}/sample-app:${VERSION} \
    --annotation com.googleapis.cloudmarketplace.product.service.name=${SERVICE_NAME}

crane mutate ${REGISTRY}/sample-app:${MINOR_VERSION} \
    --annotation com.googleapis.cloudmarketplace.product.service.name=${SERVICE_NAME}
```

Verify each tag before proceeding:

```bash
crane manifest ${REGISTRY}/py-app:${VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/py-app:${MINOR_VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/sample-app:${VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/sample-app:${MINOR_VERSION} | jq '.annotations'
```

Each should return:

```json
{
  "com.googleapis.cloudmarketplace.product.service.name": "services/gcp-mp-poc-tf-k8s.endpoints.searce-cloud-products.cloud.goog"
}
```

---

### Step 5 — Generate Metadata Files

The CFT CLI reads the Terraform source and generates the metadata files the Marketplace portal requires. Re-run this whenever `variables.tf`, `outputs.tf`, `versions.tf`, or the IAM/services files under `test/setup/` change.

```bash
# Generate metadata.yaml
cft blueprint metadata -p ./terraform -q

# Generate metadata.display.yaml
cft blueprint metadata -p ./terraform -q -d

# Validate both files
cft blueprint metadata -p ./terraform -v
```

Do not re-run CFT if only `main.tf`, `schema.yaml`, or `README.md` changed — those files do not feed into metadata generation and re-running will overwrite any manual edits to `metadata.yaml`.

---

### Step 6 — Package and Upload the Terraform Module

Marketplace downloads the Terraform module from a GCS zip at deploy time. The `test/` directory must be excluded — it is a local CFT convention and is not valid inside a Marketplace module.

```bash
export BUCKET_NAME="mp-gke-tf-modules"
export PRODUCT_NAME="sample-app"
export ZIP_NAME="${PRODUCT_NAME}-tf-module-${VERSION}.zip"

# Create and version the bucket (first time only)
gsutil mb gs://${BUCKET_NAME}
gsutil versioning set on gs://${BUCKET_NAME}

# Package the module
cd terraform/
zip -r ../scripts/${ZIP_NAME} ./* \
    -x "**/.*" \
    -x "**.terraform*" \
    -x "**/test/*"
cd ../

# Verify the zip contents before uploading
unzip -l scripts/${ZIP_NAME}
unzip -p scripts/${ZIP_NAME} variables.tf | grep goog_cm_deployment_name

# Upload to GCS
gsutil cp scripts/${ZIP_NAME} gs://${BUCKET_NAME}/${PRODUCT_NAME}/${VERSION}/${ZIP_NAME}
```

The module will be available at:
`gs://mp-gke-tf-modules/sample-app/1.0.0/sample-app-tf-module-1.0.0.zip`

---

### Step 7 — Register and Validate in the Producer Portal

1. Open the [Google Cloud Producer Portal](https://console.cloud.google.com/producer-portal/)
2. Select your publisher project and create a new product listing
3. Choose **GKE App (via Terraform)** — not GKE App via Anthos
4. Note the generated **Service Name** and use it in Step 4 above
5. Under **Container images**, register both image paths:
   ```
   us-docker.pkg.dev/searce-cloud-products/gcp-mp-poc-tf-k8s/py-app
   us-docker.pkg.dev/searce-cloud-products/gcp-mp-poc-tf-k8s/sample-app
   ```
6. Under **Deployment Package**, provide the GCS URI:
   ```
   gs://mp-gke-tf-modules/sample-app/1.0.0/sample-app-tf-module-1.0.0.zip
   ```
7. Click **Save and Validate**

## Pre-Upload Validation Checklist

Run these locally before every upload to catch the most common validation failures:

```bash
# 1. Confirm test/ is excluded and goog_cm_deployment_name is present
unzip -l scripts/${ZIP_NAME} | grep test     # should return nothing
unzip -p scripts/${ZIP_NAME} variables.tf | grep goog_cm_deployment_name

# 2. Terraform syntax check
cd terraform/
terraform init && terraform validate
cd ../

# 3. CFT metadata validation
cft blueprint metadata -p ./terraform -v

# 4. Annotation check on all four tags
crane manifest ${REGISTRY}/py-app:${VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/py-app:${MINOR_VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/sample-app:${VERSION} | jq '.annotations'
crane manifest ${REGISTRY}/sample-app:${MINOR_VERSION} | jq '.annotations'
```

## Local Testing

```bash
cd terraform/
terraform init

terraform apply \
    -var="project_id=${PROJECT_ID}" \
    -var="cluster_name=your-test-cluster" \
    -var="cluster_location=us-central1-a" \
    -var="goog_cm_deployment_name=test-deployment" \
    -var="image_repo=${REGISTRY}/py-app" \
    -var="image_tag=${VERSION}" \
    -var="helm_chart_name=sample-app" \
    -var="helm_chart_repo=${REGISTRY}/sample-app" \
    -var="helm_chart_version=${VERSION}"

# Verify deployment
kubectl get pods -n test-deployment

# Access the application
kubectl port-forward svc/test-deployment-svc -n test-deployment 8080:80
# Visit http://localhost:8080
```

## Key Design Decisions

**OCI Helm chart via `helm push`** — The chart is stored as a proper OCI artifact in Artifact Registry and pulled natively by the `helm_release` Terraform resource using `repository = "oci://..."`. This replaces the previous pattern of packaging chart files into a Docker image and extracting them with `crane` at plan time. The old approach ran a shell script during `terraform plan`, which caused authentication failures in the Marketplace portal's Cloud Build runner.

**`goog_cm_deployment_name` variable** — Required by Marketplace. The portal injects the customer's chosen deployment name into this variable at deploy time. It is used as both the Helm release name and the Kubernetes namespace name, ensuring resource names are unique across multiple deployments of the same product in a single cluster.

**Application CRD (`app.k8s.io/v1beta1`)** — Included in `chart/templates/application.yaml`. Required by Marketplace to register the deployment in the GCP console's Applications tab. Without it validation passes but the deployed app does not appear in the console.

**`test/setup/` directory** — Used only by the CFT CLI during metadata generation. It is never included in the Terraform zip uploaded to GCS. CFT reads `test/setup/iam.tf` and `test/setup/main.tf` to populate `spec.requirements.roles` and `spec.requirements.services` in `metadata.yaml`.

**`schema.yaml` image types** — The three variable types (`NAME`, `REPO_WITH_REGISTRY_WITH_NAME`, `TAG`) map to different segments of an image URI. For `us-docker.pkg.dev/project/repo/sample-app:1.0`, `NAME` resolves to `sample-app`, `REPO_WITH_REGISTRY_WITH_NAME` resolves to `us-docker.pkg.dev/project/repo/sample-app`, and `TAG` resolves to `1.0`. All three must be declared in both `schema.yaml` and `variables.tf` even if not all are used in `main.tf`.
